The version of agent-driven development that built Teide puts the agent at the wrong end of the pipeline from where most people put it.

The agent does not design. I design. I write the spec by hand: a goal, a context paragraph naming the constraint that makes the work worth doing, an architecture diagram, the fallback rules, the success criteria. Two hundred to five hundred lines, mostly prose. The first line of every spec is the contract:

```
> For Claude: REQUIRED SUB-SKILL: Use superpowers:executing-plans
> to implement this plan task-by-task.
```

The agent reads the spec and executes it. Not interprets, not extends, not improves. Each task in the spec is small enough that the agent's output can be reviewed against the spec line for line. The spec is the test.

The `docs/plans/` directory of the Teide repo is the versioned trail of these specs. The radix-partitioned join was a spec. The eight-pass optimiser was a spec. The Datalog surface was a spec. The completeness audit that sweeps for missing features was a spec. Each dated. Each with the same goal-context-architecture-success structure. None of them lets the agent invent.

It is more work to write the spec than to write the code. That is the trade. I pay a careful design tax in order to delegate the typing. For projects where the design is the hard part - a query optimiser, a graph-traversal opcode, a radix join - the trade is good. For projects where the typing is the hard part - boilerplate, glue, plumbing - it is even better.

The `CLAUDE.md` per sub-project carries the durable constraints every spec inherits without restating: no malloc, no third-party names, morsel-only processing, the `ray_` prefix. Those are not rederived in each spec; they are the project's grammar.

I did not let the agents design Teide. I would not have shipped Teide without the agents.
