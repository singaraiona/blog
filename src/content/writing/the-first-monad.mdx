Six months in. The repo's first language primitive landed back in February, the day after the initial commit. I want to write down what it took to get there, because the shape that one twenty-line function set is still the shape every kernel has now.

A monad in this language is a unary verb. The literal commit message was "introduce first monad". The primitive is `count`: it takes a column, returns its length. That is all. Twenty lines including the header.

The reason it matters is that I had to settle every structural decision to land it. How does the dispatch table look? Where does the result vector get its memory? What is the type of the function pointer? How does the parser recognise a primitive? None of these had answers when I opened the repo. By the time the first monad shipped, they did.

The dispatch is by arity, the way K does it, the way O did it in ThePlatform. Monads have their own table. Dyads will have their own table. There is no "function" type that absorbs every arity, because that flattens the cost; with arity-classed tables, the call site does no branching beyond the arity it already knows.

The result memory comes from the caller. The kernel writes into a slice the caller pre-sized. This is a rule I learned in ThePlatform and want to hold from the first kernel here. No allocation in the inner loop. The kernel does not own its output.

```c
void monad_count(block *out, const block *in) {
    out->n = 1;
    ((i64 *)out->data)[0] = in->n;
}
```

That is the whole primitive. If it does not look like much, that is exactly the point. The shape it sets - signature, memory contract, table entry - is what the next two thousand kernels will inherit. Get the shape right and the kernels are short. Get it wrong and the kernels are forever explaining themselves.
