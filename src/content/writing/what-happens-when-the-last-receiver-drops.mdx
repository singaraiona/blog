Two months of using hyperbridge inside theplatform; one bug class I had been ignoring is now closed.

The bug class is "what is the channel supposed to do when an endpoint goes away while the other end still has work".

Concretely:

- The last receiver drops. Senders continue to push values. Those values pile up in a channel that no one will ever drain. Memory leak.
- The last sender drops. Receivers wait forever on a value that will never come. Deadlock.
- A sender drops mid-transaction. Half-published values become visible. Logical corruption.

The first two are textbook. I wrote them in a notebook before I started the crate and forgot to implement them. The fix is one struct field per side - the channel knows how many senders and receivers are alive - and a check on each operation. When the last one of either side drops, the channel transitions to closed; the other side gets `Err(Closed)` on every subsequent call. No more silent leaks.

The third was subtler. A sender that crashes mid-call leaves the channel in whatever state the atomic was in when the panic landed. The fix is to do the visibility-store last, after every other operation has succeeded; if a panic happens earlier, the value was never visible. This sounds obvious. It is, in the second-edition writeup. It was not obvious when I wrote the first edition.

The other change today is a small one: senders and receivers each carry a handle to the opposite end. `sender.receiver()` returns a fresh cloned receiver. This is a convenience; it lets a piece of code that was given a sender create a fan-out structure without threading the receiver through.

Three weeks of staring at this bug class. One day of fixing it.
